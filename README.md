# unraid

An [orca](https://github.com/scottdkey/orca) plugin that adapts an Unraid host's
GraphQL API into the orca system: typed queries over array / shares / Docker /
plugins, an endpoint registry, a topology collector, and version-aware schema
drift detection.

The plugin's only orca dependency is `plugin-toolkit` — every orca primitive it
touches (contract, graphql, http, runtime, serde, schemars, clap, anyhow,
tracing, db, dispatch, derive) flows through the toolkit and its prelude.

## What it does

| Surface | What it is |
|---|---|
| **Typed client** (`src/lib.rs`) | `Client` over `plugin_toolkit::graphql::Client::query_typed` — `installed_plugins`, `array`, `shares`, `docker_containers`, `parity_history`, `add_plugin`, `remove_plugin`, `probe_version`. |
| **Endpoint registry** (`src/endpoint.rs`) | `unraid.{list,detail,create,update,delete}` — generated by `endpoint_resource!`. Each row points at one Unraid host's GraphQL API; `api_key` is stored secret-side and sent as `x-api-key`. |
| **Topology collector** (`src/topology.rs`) | Emits one `container` claim per Docker container per registered endpoint, so Unraid Docker workloads surface in the systems graph (the local Docker socket is `root:docker`-only, so GraphQL is the read path). |
| **Schema tool** (`src/tools.rs`) | `unraid.schema` — list committed schema versions, pull a fresh introspection from a live host, or report drift between live and committed schemas. |

## Version + drift detection

The typed client is codegenned at build time from committed introspection
schemas. Each committed `schemas/<version>.introspection.json` produces a
generated module. At runtime:

- `Client::new_probed` probes `vars { version }` and routes to the matching
  generated module, falling back to the newest committed version (with a
  one-time `tracing::warn`) when the live version has no committed schema.
- `Client::warn_on_drift` compares the live introspection hash against the
  committed schema and warns when they diverge.
- `unraid.schema --check_drift` surfaces the same comparison as a tool result.

## Adding a new Unraid version

Codegen is purely additive — no code edits required:

1. Pull a fresh introspection from a live host running the new version:
   ```
   unraid.schema --from https://<host> --api_key <key> --dir schemas
   ```
   (or `cargo run` the equivalent against your host). This writes
   `schemas/<version>.introspection.json`.
2. Commit the new JSON file.
3. Rebuild. `build.rs` runs `graphql_client` codegen over every committed
   schema, and `SUPPORTED_VERSIONS` / `SCHEMAS` pick up the new module
   automatically.

If a query in `queries/*.graphql` references a field that changed between
versions, codegen for the affected version will fail — adjust the query or add a
version-specific variant.

## Building

The pinned `plugin-toolkit` git dep resolves from the rc tag in `Cargo.toml`.
The committed `.cargo/config.toml` `[patch]` redirects it to a local orca
checkout at `../orca` for development; remove the patch to build purely against
the tag (as CI does).

```
cargo build
cargo test
cargo clippy --all-targets -- -D warnings
```
